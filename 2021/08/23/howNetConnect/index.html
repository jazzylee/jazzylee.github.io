协议栈要做的事情归纳为下面四个步骤

Step1:通信准备-创建套接字

初探

这个部分书上写得有点模糊，到底什么是管道，暂时还没有讲清楚

服务器启动后便创建套接字

客户端创建套接字，延伸出管道连接到服务器的套接字

套接字创建完成后，协议栈会返回应用程序一个描述符用于区分不同的套接字

详解

套接字的实体是通信控制信息(通信对象IP/port，通信操作的状态，…)协议栈需要根据这些信息判断下一步的行动(例如长时间无响应是否重发等等)

创建之后该描述符返回给应用程序，后面应用程序委托协议栈发送网络请求时提供该描述符即可对通信对象/通信状态做出区分

PS：刚创建完的套接字中还没有通信对象的信息，下一步中需要让协议栈知道这些信息

Step2:将管道连接到服务器的套接字上

初探

调用Socket库中名为connect的函数，传入描述符，IP，端口号(用于区分不同的套接字)

也许有人会说：“能不能用前面创建套接字时提到的那个描述符来识别
套接字呢？”这种方法其实是行不通的，因为描述符是和委托创建套接字的
应用程序进行交互时使用的，并不是用来告诉网络连接的另一方的，因此
另一方并不知道这个描述符。同样地，客户端也无法知道服务器上的描述
符。因此，客户端也无法通过服务器端的描述符去确定位于服务器上的某
一个套接字。所以，我们需要另外一个对客户端也同样适用的机制，而这
个机制就是端口号。

connect完成后，协议栈会保存对方的IP/port信息在套接字中

详解

在有线拨号中，有连接、通话、挂断三个阶段，于是这个词被沿用了下来

连接就是交换含控制信息；

  1. 客户端这边：将连接对象的信息告知协议栈，将会把客户端的IP/port告知服务器

  2. 服务器这边：初始状态是不知道通信对象的，需要客户端告知

控制信息分为两类：

  1. 位于网络包的开头，又叫做头部信息，整个通信过程中都需要

  2. 保存在套接字中，位于协议栈中的内存空间里，控制协议栈的操作

连接操作的实际过程

  1. 调用connect，传递给协议栈中的TCP模块
  2. 客户端的TCP模块与服务器的TCP模块交换控制信息
    • 客户端创建(要开始收发数据的)TCP头部信息，SYN置1
    • TCP委托IP模块发送
    • 数据到达服务器IP模块
    • IP->TCP
    • 服务器TCP模块根据头部信息找到端口号对应的套接字，将状态改为正在连接
    • 服务器TCP返回相应(在TCP头部设置发送方/接收方/SYN/ACK=1)
    • 服务器TCP模块委托IP木块相应数据
    • 客户端收响应应，更新套接字的状态
    • 客户端将ACK置1发回服务器
  3. 套接字进入随时可以收发数据的状态，业界习惯描述为有一根管道把两个套接字连接了起来

Step3.1:收发数据之http请求

初探

发：在内存中准备要发送的数据(http请求消息)，使用write函数(参数为描述符)收发数据；

收：使用read函数接收，参数为接受缓存的内存地址(即缓冲区)

详解

协议栈会将要发送的数据存在发送缓冲区，至于何时才发送，和下面2个因素有关

  1. MTU(Maximum Transmit Unit)

    MTU表示一个网络包的最大长度(包含头部)，以太网中一般是1500字节。减去头部的长度就是一个网络包中最大的数据长度，叫MSS(Maximum Segment Size)。

    因此对于大于MSS的数据需要拆分传输：

  2. 协议栈内部的计时器

    避免发送频率低，迟迟凑不到MTU导致长期无法发送的问题

ACK:确认收到/重传

  1. 分包发送时，会将当前这块数据位于总数据中的字节序号写在TCP头部(TCP头部信息的“序号”字段)
  2. 接收方确认相应：也就是说会将到目前为止接收数据的长度累加，写在ACK字段中
  3. 遇到错误时重传，几次重传无效后会强制结束通信，例如网络中断/服务器宕机等情形

ACK等待时间与滑动窗口

  • 网络传输拥塞时，ACK号返回得变慢，这种情况需要延长等待时间，但等待时间不是越长越好，否则会造成真正要重传的数据正确完成传输的时间过长的问题。

  • TCP采取的是根据ACK号返回的时间动态调整等待时间，ACK返回慢->延长等待时间，反之缩短。

  • 最容易想到的一来一回式的ACK管理有些浪费，采用滑动窗口方式更加高效。接收方需要告知对方自己最多能接收多少数据，以解决接收方来不及处理超量的数据导致接收缓冲区溢出的问题。

  • 接收方什么时候告知自己当前还能接收多少数据以及何时返回ACK号也是一个问题。

    分开来看，更新窗口信息发生在应用程序从接收缓冲区取出数据时就可以了；ACK号的回传发生在接收方收到没问题的数据时

    结合起来看，ACK回传可以和更新窗口在一个包里发送，减少包的数量；此外有连续的ACK或连续的窗口更新时，只需要发送最新的数据就可以。

Step3.2:收发数据之http响应

浏览器委托协议栈发送网络请求之后，就会调用read函数获取响应消息。

两个操作是连接在一起的

但是由于请求消息刚发送出去，还读不到数据，于是协议栈会将read操作挂起，等到服务器响应数据后再执行接收操作：

  1. 接收时先判断是否有数据丢失，若无，返回ACK
  2. 数据暂存到缓冲区，按顺序拼接
  3. 将数据复制到应用程序指定的内存地址中
  • 择时发送窗口更新

Step4:断开-断开管道并删除套接字

调用close函数，任意一方都可发起断开管道。发起断开的一方调用该函数之后协议栈会生成包含断开信息的TCP头部(FIN=1)，同时在套接字中也会记录断开操作的信息。另一方收到含有FIN=1的头部时，会回传一个ACK以示收到。最后，也调用close函数结束操作。

套接字不会立即删除，而是会等一段时间后再删除。

双绞线

双绞的作用

  • 抵消外部噪声

    当电磁波接触到信号线时，会沿电磁波传播的右旋方向产生电流，这种电流会导致波形发生失真。如果我们将信号线缠绕在一起，信号线就变成了螺
    旋形，其中两根信号线中产生的噪声电流方向就会相反，从而抵消，噪声得到了抑制。

  • 抑制串扰

    相邻的信号线会相互影响：由于传输的信号本身就是一种电流，当电流流过时就会向周围发出电磁波，这些电磁波对于其他信号线来说就成了噪声。

    至于是如何通过改变节距抑制噪声的，暂时还没看明白

双绞线的种类

集线器：广播信号

简单来说，就是将加入的信号从所有的输出口输出。

各PC检查目的地的MAC，如果是自己则保留。

交换机的包转发

信号到达网线接口之后，PHY模块进行接收，MAC模块将信号转为数字信息，FCS校验，无误则放入缓冲区(类似网卡做的事，但是不会检验MAC，而是保存所有的包)。

包转发需要查MAC地址表，地址表中保存了MAC地址与端口的关系。

交换机内部有很多开关，控制着输入与输出的连接(可多路并行工作)

地址表的维护

  • 有包要发送时，将发送方的端口和MAC写入表中
  • 记录并非长期有效，表项不使用后需要移除
  • 重启可以清空表

路由器的包转发

路由器内部可以简单分为转发模块和端口模块两部分。转发模块委托端口模块收发数据包(这种关系很像计算机中IP模块委托网卡)。

在接收数据时，首先检验FCS序列，如果没问题，再看接收MAC是不是自己，如果是，则查表转发。

MAC 头 部的作用就是将包送达路由器，其中的接收方 MAC 地址就是路由器端口 的 MAC 地址。因此，当包到达路由器之后，MAC 头部的任务就完成了， 于是 MAC 头部就会被丢弃

交换机查表时最多值匹配一条记录，路由器匹配一个网段(像极了者车站候车|去xxx方向的旅客请到yyy检票口检票)

在转发包的时候，带上自己的MAC地址

● 如果路由表的网关列内容为 IP 地址，则该地址就是下一个转 发目标

● 如果路由表的网关列内容为空，则 IP 头部中的接收方 IP 地址 就是下一个转发目标。这时候还能用ARP查到对方的MAC吗？

关于路由器和交换机的区别：

简单来说，IP （路由器）负责将包发送给通信对象这一整体过程，而其中将包传输到下一 个路由器的过程则是由以太网（交换机）来负责的

地址转换的原理

维护一个地址转换表，使用不同的(TCP/UDP)端口号来区别内网中的 不同终端。

端口号有16位，因此一个公网IP可以虚拟看作几万个私网IP

着对于没有在访问互联网的内网设备，是无法 从互联网向其发送网络包的，只要事先手动添加这样的记录就可以了。

路由器的包过滤

在对包进行转发时，根据 MAC 头部、IP 头部、TCP 头部的内容 ，按照 事先设置好的规则决定是转发这个包，还是丢弃这个包

接入网&运营商

汪洋大海，浩浩汤汤。

接入网就是指连接互联网与家庭、公司网络的通信线路。家用的接入网方式包括 ADSL(Asymmetric Digital Subscriber Line)、FTTH、CATV、电话线、ISDN 等，公司 则还可能使用专线。

ADSL

MODEM

ADSL Modem 将包拆分成ATM信元，并转换成电信号(使用QAM调制)发送给分离器。

ADSL 使用间隔为 4.3125 kHz 的上百 个不同频率的波进行合成

当 Modem 通电后，会 发送测试信号，并根据信号的接收情况判断使用的频段数量和每个频段的 比特数，这个过程称为训练（握手），需要几秒到几十秒的时间。

分离器

分离器其实就是个filter，当信号从电话线传入，分离器负责将ADSL信号和话音信号(低频)分离。

PPP与隧道

PPP 协议：传统电话拨号上网时使用

PPPoE：以太网的点对点协议

服务器端的局域网

有关防火墙、缓存服务器、负载均衡器等等

防火墙

专用的硬件或软件，作为路由器包转发规则 pro plus版本。

作用是只允许发往指定服务器的指定应用程序的网络包通过，从而屏蔽其他不允许通过的包。

分布式架构-负载均衡

使用负载均衡器的IP代替Web服务器实际地址注册到DNS服务器上，由负载均衡器决定将请求转发给哪台Web服务器。

缓存服务器-反向代理/转发/缓存

缓存服务器也需要代替Web服务器的真实地址注册到DNS服务器中。

一台缓存服务器可以对应多个Web服务器。

无缓存的情形：

客户端发消息->代理服务器接收->转发给Web服务器，HTTP请求中添加Via字段表示是转发的->Web服务器返回数据给缓存服务器->缓存服务器保存并响应给客户端

有缓存的情形：

客户端发消息->代理服务器接收->询问Web服务器消息是否有改变>304未改变|改变后的内容->(缓存服务器保存并)响应给客户端

包收发部分，客户端和服务器没什么区别

将信号转换成数字信号->FCS校验->检验接收方的MAC是否为自己->存到网卡缓冲区->网卡产生中断信号->CPU参与处理->检查IP地址、是否分片、协议号(如0x06 TCP)

数据处理到TCP这部分时，如果收到的是发起连接的包，则 TCP 模块会（1）确认TCP 头 部的控制位 SYN；（2）检查接收方端口号；（3）为相应的等待连 接套接字复制一个新的副本；（4）记录发送方 IP 地址和端口号等信息。

收到数据包时，TCP 模块会（1）根据收到的包的发送方 IP 地址、发送方端口号、接收方 IP 地址、接收方端口号找到相对应的套接字；（2）将数据块拼合起来并保存在接收缓冲区中；（3）向客户端返回 ACK。
